# Evaluation Guide

このファイルは、PHP-Studyの課題を評価するときの基準です。

評価の目的は点数を付けることではなく、ユーザーが「最低要件を満たすコード」と「意味のあるシンプルで良いコード」の違いを理解できるようにすることです。

## 評価の基本ルール

- 既存の教材ファイルは勝手に編集しない。
- 評価時も、完成コードをすぐに出さない。
- 問題文、現在のコード、実行結果、エラー文を根拠にする。
- 「動くか」と「良いコードか」を分けて見る。
- 初心者の学習中コードとして評価し、過度な最適化や高度な書き方は求めない。
- 評価した場合は、回答前に `learning-memory/evaluation-log.md` へ記録する。
- 評価ログにも完成コードや最終解答そのものは残さない。

## 評価ランク

| ランク | 意味 |
|---|---|
| S | 最低要件を満たし、シンプルで読みやすく、境界条件も意識できている |
| A | 最低要件を満たし、十分に良い。細かい改善余地がある |
| B | おおむね方向性は合っているが、一部の条件・出力・境界値に不安がある |
| C | 構文エラーはないが、問題文の要件を満たしていない |
| D | 構文エラーや実行時エラーがあり、まず実行可能にする必要がある |
| 未着手 | TODOや空関数が残っており、評価前の状態 |

## 評価観点

### 1. 実行可能性

- `php -l ファイル名.php` で構文エラーがない。
- 実行時エラーや警告が出ない。
- テストコードがある課題では、既存テストがOKになる。

### 2. 最低要件

- 問題文どおりの条件を満たしている。
- 指定された構文を使っている。
- 指定された出力文字列と意味が合っている。
- 余計な出力が混ざっていない。
- 関数問題では、`echo` ではなく必要な値を `return` している。

### 3. エンジニア視点

- 条件式が問題文と対応していて読みやすい。
- 余計な変数や処理が少ない。
- 似た処理を無理に複雑化していない。
- テスト値を変えても破綻しにくい。
- 境界条件を意識できている。

## ワーク別の最低要件

### work1: 条件分岐

- `if`, `else`, `elseif` を問題に合わせて使う。
- 比較演算子を正しく使う。
- 「より大きい」「以上」「未満」「以下」を区別する。
- 偶数・倍数では、割り算の答えではなく余りを見る。
- 指定された文字列だけを表示する。

### work2: ループ・配列

- 問題で指定された `for` または `foreach` を使う。
- ループの開始値、終了条件、増減が正しい。
- 合計、フィルタ、評価などの処理がループの中で自然に読める。
- 配列の値、キー、連想配列の扱いを混同しない。

### work3: 関数

- 関数が必要な引数を受け取る。
- 結果を `return` する。
- 既存のテストコードを変更しない。
- すべてのテストがOKになる。
- 関数の中だけで完結する処理にする。

### work4: 型宣言

- 引数と戻り値に適切な型を付ける。
- `declare(strict_types=1)` がある課題では、型の違いによる挙動を意識する。
- 型エラーを無理な回避で隠さない。
- 問題で意図された変換は、必要な場所だけで行う。

### work5: クラス基礎

- フィールド、コンストラクタ、getter、setterに型宣言がある。
- `private` フィールドを外から直接触らない。
- バリデーションで不正値を防ぐ。
- `getOlder` は既存の `setAge` の考え方を活用できている。
- `main.php` の確認用コードと整合している。

### workEx1: クラス応用

- コンストラクタで初期状態を正しく作る。
- `reload`, `fire`, 拡張マガジンの状態変化が一貫している。
- 残弾数や最大装弾数が不自然な値にならない。
- すでに満タン、残弾0、拡張マガジン未装着などの分岐を扱う。
- 状態を表す変数名が読みやすい。

### workEx2: 総合問題

- `calcAverage`, `getGrade`, `getPassedStudents` の役割が分かれている。
- `getPassedStudents` で `calcAverage` を活用している。
- 成績判定の境界値が正しい。
- 表示用関数をむやみに変更しない。
- 期待される実行結果と整合する。

## 評価コメントの型

評価するときは、次の順で短く返す。

```text
評価:
- ランク: ...
- 最低要件: ...
- エンジニア視点: ...
- 次に見る1点: ...
```

## ヒントの出し方

- D評価の場合: まず構文エラーや実行時エラーの読み方を説明する。
- C評価の場合: 問題文とコードの条件がずれている箇所を示す。
- B評価の場合: 境界条件やテスト値を追加で考えさせる。
- A評価の場合: より読みやすくする小さな改善を提案する。
- S評価の場合: どこが良いかを言語化し、次の課題へ進む判断をする。
