# Learning Memory Update Rules

Codexが学習状態を更新するときのルールです。

## 更新してよいファイル

- `LEARNING_MEMORY.md`
- `learning-memory/interaction-log.md`
- `learning-memory/evaluation-log.md`
- `learning-memory/session-template.md` を参考にした新しい記録
- `learning-memory/` 配下の新規ファイル

## 更新しないファイル

- `README.md`
- `sample/` 配下
- `works/` 配下
- 通常の学習相談では `AGENTS.md`
- 通常の学習相談では `docs/learning/` 配下

システム構築を明確に依頼された場合だけ、`AGENTS.md` や `docs/learning/` 配下を更新してよい。

## 記録する内容

システム構築以外の学習相談後は、原則として `learning-memory/interaction-log.md` に次を記録する。

- 日付
- 質問の種類
- 対象ファイル
- 質問の要約
- Codexが返した支援内容の要約
- つまずき
- 理解できていること
- 次回の支援方針

学習状態が変わった場合は、`LEARNING_MEMORY.md` に次を反映する。

- どの課題に取り組んだか
- どこまで自分で考えたか
- どの概念につまずいたか
- 何が理解できたか
- まだ曖昧なことは何か
- 次回Codexがどう助けるべきか
- 評価上、最低要件とエンジニア視点で何を次に見るべきか

評価後は、原則として `learning-memory/evaluation-log.md` に次を反映する。

- 評価したファイル
- 評価ランク
- 最低要件を満たしているか
- エンジニア視点で良い点と改善点
- 次に確認する1点

## 書き方

- 断定しすぎない。
- コードの正解そのものを書かない。
- 「観察した事実」と「推測」を分ける。
- 次回のヒント方針がわかるように書く。
- 完成コードや最終解答をそのまま残さない。
- 1回の記録は短く、後から見返して次の支援に使える内容にする。

## 更新タイミング

次の種類の依頼では、ユーザーから追加確認を取らずに記録する。

- 課題のヒント依頼
- エラー相談
- 考え方の相談
- 答え合わせ
- 評価依頼
- 実行結果の読み取り

次のように言われた場合も更新する。

- `学習ログを更新して`
- `今の理解度を記録して`
- `次回のためにメモして`
- `LEARNING_MEMORY.md を更新して`
- `評価ログを更新して`
- `この課題の評価を記録して`

次の依頼は通常の学習ログには記録しない。

- `AGENTS.md` の変更
- `LEARNING_MEMORY.md` の構造変更
- `docs/learning/` や `learning-memory/` の仕組み作り
- メモリーや評価システム自体の設計相談
